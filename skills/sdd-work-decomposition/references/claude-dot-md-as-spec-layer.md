# CLAUDE.md as Spec Layer

How CLAUDE.md / AGENTS.md serves as the specification carrier between SDD phases and the implementation agent.

## The Handoff Problem

Between SDD phases, knowledge must transfer:
- Spec authoring knows what the system should do
- Work decomposition knows how the tasks are sequenced
- Implementation agent knows how to build things

These three contexts are held by different agents at different times. CLAUDE.md is the artifact that carries the spec into the implementation phase without requiring the implementation agent to re-read the entire SDD pipeline.

## What CLAUDE.md Carries

The CLAUDE.md generated by `sdd-work-decomposition` should include:

### 1. Active Task Scope
Which task(s) the implementation agent is currently executing. Only one task's scope should be in a given CLAUDE.md — the implementation agent should never see a CLAUDE.md that describes 12 tasks when it's supposed to implement task 3 of 12.

### 2. Acceptance Criteria for the Current Task
The specific ACs from SPEC.md that this task must satisfy. These are the implementation agent's primary directive — if the code doesn't satisfy the ACs, the task is not complete regardless of how well-written the code is.

### 3. Architecture Constraints from the Spec
Which SPEC.md sections define the architecture and contracts that the implementation agent must honor. Include: interface signatures, data schemas, protocol definitions, configuration conventions.

### 4. The Escalation Protocol
What the implementation agent should do when it encounters ambiguity. The standard SDD escalation: "If a requirement is unclear, do not proceed. Flag the ambiguity and stop." CLAUDE.md must encode this explicitly — LLMs default to forging ahead with their best guess.

### 5. Commit and Verification Rules
- One atomic commit per task
- Commit message must reference the task ID and spec AC
- No task is complete until its ACs can be verified

## AGENTS.md for Delegation

When the implementation agent is running as a subagent or delegate, AGENTS.md serves the same role but is optimized for agent consumption:

```
# AGENTS.md — Implementation Agent

## Your Task
Implement T-003: User Registration Endpoint

## Acceptance Criteria (from SPEC.md US-002)
1. [AC-002.1] POST /register accepts email + password
2. [AC-002.2] Returns 201 with user profile on success
3. [AC-002.3] Returns 409 if email already registered
4. [AC-002.4] Returns 422 on invalid email format

## Contracts
POST /api/v1/register -> { email: string, password: string }
Response 201 -> { id: string, email: string, created: datetime }
Response 409 -> { error: "email_already_registered" }
Response 422 -> { error: "invalid_email" }

## Constraints
- Password min 8 chars, stored as bcrypt hash
- Email validated by regex before DB query
- Rate limit: 10 req/min per IP

## Escalation
If any AC, contract, or constraint is ambiguous or contradictory:
DO NOT PROCEED. Flag the issue and return to the orchestrator.
```
