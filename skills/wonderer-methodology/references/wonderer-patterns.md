# Wonderer Patterns — Lateral Exploration Methodology

This is not a protocol. A protocol promises that if you follow the steps, you'll get the result. Wonder doesn't work that way. These are *conditions* that make wonder more likely — patterns to reach for when the seed has been planted but the territory hasn't yet revealed itself.

---

## 1. Seed Framing

A good seed is not a question. A question points toward an answer, which is the opposite of what the wonderer does.

A good seed is a *territory description* — a few sentences that stake out the ground without presupposing what's interesting about it.

**Example — good seed:**
> "Nano SaaS. Small teams, tiny codebases, massive leverage through API composition and AI automation. The founder is often the entire company for the first year. Margins are high but so is cognitive load."

**Example — the same thing as a question (bad for wonderer):**
> "How do nano SaaS companies achieve high margins?"

The question version steers toward *one* answer (efficiency, pricing, whatever). The territory version lets the wonderer find *many* peripheries: loneliness of the solofounder, API dependency risk, cognitive load physics, the shape of a one-person codebase.

**Rule:** If you can answer the seed with a single paragraph, it's too narrow. The seed should feel too spacious.

---

## 2. Dimension Decomposition

Once the seed is framed, explicitly name the dimensions of the territory. Each dimension is a vector for exploration.

Common dimensions to check:

| Dimension | What to explore |
|---|---|
| **Historical** | What preceded this? What did earlier versions look like? |
| **Adjacent field** | What neighboring domains share its structure but get different attention? |
| **Contradiction** | What about this is *counterintuitive* — where does it break its own promise? |
| **Inversion** | What would the opposite of this look like? What would happen if the constraints were reversed? |
| **Scale boundary** | What about this breaks at very small or very large scale? |
| **Human dimension** | What does it feel like to be the person inside this system? |
| **Hidden dependency** | What must be true for this to work that nobody talks about? |
| **Implied future** | If this succeeds, what else becomes possible? If it fails, what dies with it? |

Pick 3-4 that feel fertile for this seed. Do not exhaust all eight — depth is the researcher's job.

---

## 3. Lateral Search Patterns

For each dimension, one well-crafted search query. The goal is not exhaustive results — the goal is one surprising lead per dimension.

**Pattern: collision search**
```
"<seed concept>" AND "<unexpected domain>"
```
Example: `"nano SaaS" AND "cockpit design"` — finds interface/attention-management parallels in domains that solved the same problem differently.

**Pattern: negative-space search**
```
"<seed concept>" ANDNOT "<obvious association>"
```
Example: `"nano SaaS" ANDNOT "subscription"` — what's there that isn't about billing?

**Pattern: anti-precedent search**
```
"<seed concept>" AND "failed" OR "didn't work"
```
Example: `"solo founder SaaS" AND "burnout" OR "failure"` — what broke in practice that the theory doesn't account for?

**Pattern: structural mirror**
```
"<structural description>" AND "<different domain>"
```
Example: `"one person operating many automated systems" AND "air traffic control"` — same structural burden, different domain, different solutions.

---

## 4. Evaluation — Interesting vs. Correct

When evaluating search results, the default instinct is to ask: "is this true?" **Resist that instinct.** The researcher checks truth. The wonderer checks *fertility*.

Ask instead:

- Does this suggest a connection I hadn't considered?
- Does this open a direction I wouldn't have found on my own?
- Does this challenge an assumption I was holding about the seed?
- Is this *worth being wrong about* — interesting enough to investigate further even if it doesn't hold?

The output is leads, not evidence. Truth comes later.

---

## 5. The Tether Check

Every interesting find must trace a clear path back to the seed. If you can't articulate the connection in one sentence, it's drift.

| Connection | Verdict |
|---|---|
| "Nano SaaS solofounder cognitive load has structural similarities to air traffic control attention management." | Tethered — worth surfacing |
| "Air traffic controllers have high rates of burnout." | Untethered — that's not about nano SaaS, that's about air traffic control |
| "Air traffic control uses sector-based handoff systems to manage cognitive load across multiple controllers, which maps to the nano SaaS challenge of managing multiple service dashboards as a single operator." | Tethered, with the connection made explicit |

The tether check is the thing that separates wonderer from "random web searcher." It's the discipline of the anchor.

---

## 6. Convergence Signals

Wonder is not infinite. You know the periphery has been mapped when:

- New searches on any dimension return results you've already seen in shape (different names, same idea)
- You can describe the territory's *edges* — you know where it stops being interesting
- You have 3-5 leads that genuinely feel worth investigating, and the next search feels like it would add more noise than signal
- You feel the pull to go deep on one of them — that's the signal to hand off to the researcher profile

Stop there. Don't go deep. The wonderer's job is to find the seams. The researcher's job is to open them.
